# Two-Tier Memory Summaries

## Overview

Currently, each memorized item stores a single `summary_text` field that serves as both the search target and the content returned to the AI model. This creates a trade-off: a short summary is good for search metadata but too thin to give an AI model meaningful context; a detailed summary gives rich context but adds noise to the embedding space. This feature introduces a two-tier summary structure — a short **headline summary** used as the embedding target (optimised for semantic search), and a richer **detail summary** that is retrieved and returned to the model once the headline matches a query.

## User Story

As a user, I want each memory to have a concise title-like summary for search and a detailed narrative summary for retrieval, so that semantic search finds the right memory and the AI model receives enough context to actually use it.

## Scope

### In Scope

- Add a `detail_summary` field to the `memorized_items` table alongside the existing `summary_text` (which becomes the headline/search summary)
- Update the draft approval flow so the extension generates both a headline and a detail summary before submitting to the backend
- Update `DraftCreate` and `DraftApprove` schemas to carry both summary fields
- Embed the headline summary (not the detail) for vector search — keeps the embedding space clean
- Return the detail summary (not the headline) in `ContextResponse.relevant_memories` so the AI model receives rich context
- Update the `MemoryResult` schema to expose both fields
- Update the frontend draft review card to show and allow editing of both summaries
- Update the memories list / detail view to display both fields

### Out of Scope

- Changing the embedding model or vector dimensions
- Re-embedding existing memories (existing rows get `detail_summary = NULL`; the headline `summary_text` is used as fallback)
- Multi-level summarisation (more than two tiers)
- Automatic re-summarisation when a memory is edited

## Functional Requirements

1. `memorized_items.summary_text` is repurposed as the **headline summary** — a single sentence (≤ 200 chars) suitable for embedding and search metadata.
2. A new `memorized_items.detail_summary` column (Text, nullable) stores a **detail summary** — multiple sentences describing what was discussed, key decisions, entities, and any facts extracted.
3. The Chrome extension summarisation prompt is updated to return a JSON object with both `headline` and `detail` fields.
4. `DraftCreate` gains an optional `detail_summary` field; the existing `summary_text` field maps to the headline.
5. `DraftApprove` gains an optional `detail_summary` field; if omitted, `detail_summary` is set to `NULL` on the resulting `MemorizedItem`.
6. `EmbeddingService` always embeds `summary_text` (headline) regardless of `save_mode`, except when `save_mode = full_conversation` where it falls back to the full text as before.
7. `ContextResponse.relevant_memories` returns `detail_summary` (if present) as the primary content field; falls back to `summary_text` if `detail_summary` is NULL.
8. The draft review card in the frontend shows two editable text areas: "Headline" (single line) and "Detail" (multi-line).
9. The memory detail view displays both the headline and the detail summary.
10. Existing memories with `detail_summary = NULL` are displayed gracefully (headline only).

## UI / UX Notes

- **Draft review card**: Two stacked fields — a compact single-line "Headline" input above a taller "Details" textarea. Labels clarify: "Headline (used for search)" and "Details (returned to AI)".
- **Memory list card**: Shows only the headline as the card title. Detail summary is shown on expand or in the detail view.
- **Memory detail page**: Shows headline prominently, detail summary in a collapsible or secondary section below.
- **Semantic search results**: The similarity badge and card title use the headline; the snippet shown under the title uses the first sentence of the detail summary.

## Data / API Changes

### New DB column

| Table | Column | Type | Nullable | Default |
|---|---|---|---|---|
| `memorized_items` | `detail_summary` | Text | Yes | NULL |

### Schema changes

- `MemoryDraftRead`, `DraftCreate`, `DraftApprove` — add `detail_summary: str | None`
- `MemorizedItemRead` — add `detail_summary: str | None`
- `MemoryResult` (context response) — rename `summary` → `headline`; add `detail: str | None`

### Endpoints affected

- `POST /api/v1/drafts/` — accepts `detail_summary`
- `POST /api/v1/drafts/{id}/approve` — accepts `detail_summary`, writes to `MemorizedItem`
- `GET /api/v1/memories/` and `GET /api/v1/memories/{id}` — return `detail_summary`
- `POST /api/v1/context/query` — returns `headline` + `detail` per result

## Privacy & Security Considerations

The detail summary is generated by the Chrome extension using the user's own model API key — the same flow as today. It never contains raw conversation text. The extension's model call produces both the headline and the detail before anything touches the vault backend. The `DraftCreate` schema continues to have no `raw_text` field, enforcing the core privacy rule at the API boundary.

## Open Questions

- Should the detail summary have a length cap enforced by the backend (e.g. 2000 chars), or is it left unconstrained?
- Should the extension be given explicit guidance in the prompt about headline length (e.g. "under 15 words") to keep embeddings consistent?
- For existing memories without a detail summary, should the UI offer a one-click "Generate detail" action that calls the model API through the extension?
- Should semantic search rank results using headline embedding distance only, or should a future iteration also embed the detail summary separately?
